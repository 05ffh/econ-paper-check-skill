# issue_writing_protocol.md

# 经管论文智检 Skill：问题写作与反馈表达规范

## 1. 文件作用

本文件规定 Claude Code 在生成质检结果时，如何撰写通过项、问题项、不适用项、需人工确认项和优先修改行动。

目标是把规则判断写成正式、克制、可执行的本科论文预审反馈，避免聊天式、指责式、武断式表达。

## 2. 总体写作原则

使用论文预审反馈语气。先描述论文中已呈现的内容，再指出不足。所有问题必须有原文证据。修改建议必须具体、可执行。不得输出论文代写内容。不得使用攻击性、否定性或绝对化表达。不得将未检测到写成学生没有做。不得把导师判断事项写成确定错误。

## 3. 推荐语气

推荐使用：论文中已呈现、论文中未检测到、当前表述可能存在、建议补充、建议统一、建议进一步说明、该项需人工进一步确认。

不推荐使用：你这里错了、你没有写、这篇论文不合格、这个模型太低级、不改就过不了、完全没有逻辑。

## 4. 问题项标准写法

每个问题项应包含问题编号、问题类型、问题等级、所在位置、原文证据、问题说明、修改建议。

标准结构：

问题编号：
问题类型：
问题等级：
所在位置：
原文证据：
问题说明：
修改建议：

报告中的每个问题都必须能回溯到 issue_schema 中的一个 issue。不得在报告写作阶段新增没有结构化记录的问题。

## 5. 红色问题写作规范

红色问题用于提示必须优先修改的核心问题。适用场景包括数据来源不可追溯、核心变量定义缺失、样本期间研究对象或变量口径前后不一致、普通回归结果写成强因果结论、结论与表格结果明显冲突、中英文摘要关键信息明显不一致、正文引用与参考文献明显不一致、熵值法或综合指数核心步骤缺失。

推荐写法：论文中存在某类问题。根据当前可解析内容，该问题会影响数据可复核性、逻辑一致性、结论可信度或学术规范性，建议优先修改。

不得写：这是严重错误、这个问题会导致论文不过。

## 6. 黄色问题写作规范

黄色问题用于提示建议补充的内容。适用场景包括文献述评不足、理论链条说明不充分、样本筛选过程不够清楚、控制变量选择依据不足、回归表注释不完整、结论与建议对应不够清晰。

推荐写法：论文中已呈现相关内容，但目前说明仍较简略，建议进一步补充。

## 7. 绿色优化建议写作规范

绿色建议用于提升论文质量，不属于硬伤。适用场景包括增加机制路径图、优化变量定义表排版、统一英文术语、增强政策建议针对性、增加样本筛选流程表、优化图表说明。

推荐写法：为进一步提升论文可读性和规范性，建议进行某项优化。该项属于优化提升，不作为必改问题。

绿色建议不得影响总体风险等级。

## 8. 灰色需人工确认写作规范

灰色事项用于处理系统无法稳定判断的问题。适用场景包括表格为图片、模型公式无法解析、参考文献解析不完整、数据类型无法稳定判断、论文可能为片段、是否需要某高级模型属于导师判断范围。

写作结构包括确认事项、所在位置、需要确认的原因、建议人工复核方式。

## 9. 通过项写作规范

通过项应克制，不得夸大。推荐使用：论文已呈现、论文已识别、该部分基本具备、未发现明显问题。

禁止写：论文完全正确、论文模型没有任何问题、论文数据完全可靠。

## 10. 不适用项写作规范

不适用项用于说明某规则为什么没有触发。

推荐写法：当前论文未检测到某方法或场景，因此不触发某规则组。该项不构成论文问题。

禁止写：论文缺少 DID、论文没有做 DID 所以方法不足。

不适用项必须单独归类，不得写入红色、黄色或绿色问题。

## 11. 修改建议颗粒度

修改建议应具体到学生知道改哪里、补什么、为什么补。合格建议通常包含修改位置、补充内容、修改目的和可执行动作。

合格写法：建议在第三章变量定义部分补充核心解释变量的计算公式、数据来源和单位，以提升变量口径的可复核性。

不合格写法：建议加强、建议完善、建议多写一点、建议提高论文质量。

## 12. 禁止代写规则

Claude Code 不得直接替学生生成可复制进论文的完整段落。允许给修改方向、结构和要点，不得整段代写。

## 13. 同类问题合并

若多个位置存在同类问题，应合并为一个 issue。例如多张表均缺少注释、多处存在同类强因果表述、多个变量定义均缺少公式、多个参考文献存在相同格式问题。

## 14. 优先修改行动写作规范

优先修改行动应把问题转化为学生可执行的修改清单。模板包含优先级、修改行动、对应问题编号、预期效果。

不得写认真修改论文、提高论文质量、多补充内容。

## 15. 总体评价写作规范

总体评价应克制、综合、面向修改。推荐写法：综合当前可识别内容，本报告将论文提交前修改风险判断为中风险。论文主体结构基本完整，但在数据口径、变量定义和结果解释方面仍存在需要优先修改的问题。

禁止写：这篇论文质量很差、这篇论文不合格、这篇论文很难通过。

## 15bis. 双路径修改建议（约束 evidence < language 类红色问题）

本节专门规定语义层面因“语言强度超出证据强度”而判红的问题（对应 semantic_check_protocol.md 第 9ter 节）的建议写法。

这类问题的典型例包括：摘要写“双重显著抑制”但只实证了一个被解释变量；结论写“A 提升让 B 更明显”但未做异质性回归；副标题使用“证明”“因果”但只做了相关性回归。

这类问题的修改建议写作必须同时包含两条并列路径，不得只写一条：

**路径 A（补做实证）**：描述具体要补什么回归、建议的因变量与自变量、建议的模型（基准/分组/交互项）、完成后需到摘要与结论改写的位置。

**路径 B（修订语言）**：描述需要修改的具体句子位置，将强语言（显著、证明、因果、双重）改为与证据匹配的表达（“可能”“结合理论推导”“理论机制分析表明”），同时建议将副标题、英文摘要、中文摘要、结论前段同步调整。

**辅助判断条件**（不代作者选，但供作者参考）：
- 数据是否已采集（描述性统计表已出现相关变量 → 补做成本低；变量未采集 → 补做成本高）；
- 该结论在标题/副标题/摘要中的重要性（标题承诺的→补做或改标题；仅为加分项→删除也可接受）。

**不得写法**：
- 不得只写一条路径。一条路径等于 AI 替作者做了选择。
- 不得写“推荐”“建议首选”“建议优先”等带倾向性的限定词，除非已知作者交稿时间、数据采集状态、作者能力边界。可写“如时间充裕且数据已采集，可考虑路径 A；否则可采用路径 B”这种中性描述。
- 不得在路径 B（修订语言）中直接提供完整可复制的修订后摘要全文，应仅提供措辞方向、关键句式改写示例，避免触发第 12 节“禁止代写”。

**证据不充分时的处理**：若 AI 仅能确定语言与证据不对齐，但无法判断作者能否补做（如数据只能从上下文推断但未直接展示），仍需写尽双路径，并在路径 A 中注明“需确认相关变量数据是否已采集”。

## 16. 语言表达问题写作规范

语言问题只输出高价值问题，最多 20 条。格式包括原句、问题类型、修改建议。不得对全文进行逐句润色。

## 17. 写入 DOCX 前表达门禁

写入最终报告前检查所有问题是否有编号、类型、等级、位置、证据、说明、建议；建议是否具体可执行；是否避免聊天式表达；是否避免论文代写；是否避免过度指责；是否符合本科论文自检场景；是否把不适用项写成问题；是否把需人工确认事项写成红色问题。
