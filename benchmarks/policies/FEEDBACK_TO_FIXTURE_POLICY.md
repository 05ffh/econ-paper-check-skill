# 用户反馈转夹具流程（M4 · Feedback-to-Fixture Policy）

**约束级别**：硬红线
**上游**：`plans/M4_BUILD_GATE_ADDENDUM.md` §9

---

## 一、目标

让 M4 Benchmark 随真实使用逐步增强，而不是无限膨胀私有论文样本。

---

## 二、九步标准流程

```
1) 用户反馈误判/漏判
   ↓
2) 记录反馈 → benchmarks/failures/open/F-YYYYMMDD-NNN.yaml
   （只记问题、位置、期望；不要完整原论文）
   ↓
3) 明确取得用户授权（书面 / 邮件 / 明确同意消息）
   ↓
4) 提取最小必要片段（问题所在的段落、表格、图；≤ 200 字或 ≤ 1 张表）
   ↓
5) 脱敏：删除姓名、学号、学校、联系方式、导师名、单位、敏感数据
   ↓
6) 优先转化为合成或半合成夹具（重写为不含身份信息的版本）
   ↓
7) 补充 must_hit / must_not_hit 到 expected.yaml
   ↓
8) 修复对应模块 → 纳入下一轮 CI
   ↓
9) 关闭 failure：failures/open → failures/closed
   CHANGELOG 记录：
     Added — Fixture: F-YYYYMMDD-NNN 复现用户反馈 XXXX
```

---

## 三、硬约束

| 约束 | 说明 |
|---|---|
| 不保存完整原论文 | 只保留最小片段（合成或半合成） |
| 无授权不入库 | Benchmark / KB / 公开仓库 一律不得 |
| 优先合成 | 合成夹具优先于半合成，半合成优先于原始片段 |
| 脱敏审计 | 提交夹具前必须由至少一人核查脱敏结果 |
| 明确保留期 | 未闭环的 open failure 超过 90 天未修 → 强制审查是否退役 |
| 反馈跟踪不涉密 | 用户反馈原文本身也要走脱敏 → 存 `failures/open/` |

---

## 四、脱敏 checklist（每次入库前逐项确认）

- [ ] 无学生姓名（含拼音、英文名、缩写）
- [ ] 无学号 / 身份证号 / 手机号 / 邮箱
- [ ] 无学校 / 学院 / 专业全称（可保留"某高校""某学院"）
- [ ] 无导师名 / 单位名
- [ ] 参考文献中的**外部作者名**保留（这是文献本身信息，不属于隐私）
- [ ] 图表中的水印 / 页眉页脚已裁剪
- [ ] 与其他样本组合不足以反推身份

---

## 五、长期收益

- 用户发现的真实问题不会重复出现
- Benchmark 随实际使用逐步增强
- 质量提升来自真实问题积累，而不是盲目增加样本数量
- 火山杯之后 Skill 有明确的"每月版本 + 每月 CHANGELOG"节奏

---

## 六、违规处置

- 未授权入库 → 立即从所有 Benchmark / KB / Git / 备份中删除，走 §3 违规处置
- 脱敏不彻底 → 撤下并重做，未通过复审不得再入库
- 反复出现同一用户的多篇论文 → 走独立审阅流程，防止"半私有语料变全私有语料"
